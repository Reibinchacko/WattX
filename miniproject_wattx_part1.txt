ABSTRACT
------------------------------------------------------------

WattX is an Internet of Things (IoT) based smart energy monitoring system
designed to provide domestic consumers with real-time visibility and control
over their electricity consumption. The system combines an IoT smart energy
meter, a secure cloud backend, and a user-friendly Android mobile
application. The hardware unit, built around an ESP8266 microcontroller and
PZEM-004T energy sensor, continuously measures electrical parameters such as
voltage (V), current (A), active power (kW), power factor, and frequency,
and publishes these readings to an MQTT broker at fixed intervals.

On the software side, the Flutter-based WattX mobile application connects to
Firebase Realtime Database, where the live meter readings are stored by a
Python MQTT bridge. The app visualizes this data through an interactive
dashboard and analytics module, enabling users to monitor instantaneous
consumption as well as daily, weekly, monthly, and yearly energy trends.
In addition to monitoring, WattX allows remote control of selected home
appliances (lights and fans) through relay outputs on the IoT meter; any
change initiated from the app is reflected in Firebase and propagated back
to the hardware in near real time. Smart alerts are generated for anomaly
conditions, such as sudden spikes compared to a 7‑day rolling average,
crossing user-defined budget thresholds, or prolonged idle usage.

The system further integrates a basic billing and payments module that
converts measured kWh into an estimated bill using configurable tariff
slabs and supports online bill payment through the Razorpay payment
gateway. Role-based access control is implemented to differentiate between
Regular Users, KSEB Officers, and Administrators, each with dedicated
dashboards for monitoring, complaint management, notice publishing, and
user administration. By leveraging Firebase Authentication, Firebase
Realtime Database, and Cloud Firestore, WattX achieves secure, scalable,
and low-cost deployment without the need for on-premise servers.

This mini project demonstrates the end-to-end implementation of a complete
IoT stack – from field-level sensing and MQTT-based data acquisition to
cloud storage, mobile UI, and digital payments. The proposed solution aims
to reduce electricity wastage, prevent bill shocks through timely alerts,
and improve consumer convenience by unifying monitoring, control, billing,
and complaint handling in a single application tailored for KSEB domestic
consumers.

1.1 PROJECT OVERVIEW

WattX is a smart energy monitoring system that leverages IoT and mobile
technology to give homeowners real-time visibility and control over their
electricity consumption. The application consists of two key components:

(1) An IoT smart energy meter (hardware) that measures electrical parameters
    such as power (kW), voltage (V), current (A), power factor, and frequency
    in real time using an ESP8266 microcontroller and PZEM-004T sensor.

(2) A Flutter-based Android mobile application that connects to the IoT
    device via Firebase Realtime Database and MQTT protocol to display live
    data, provide analytics, enable device control, and send smart alerts.

The system supports three types of users:
- Regular Users: Monitor energy, control appliances, view bills, receive alerts
- KSEB Officers: Manage consumer complaints, service requests, post notices
- Admins: Manage all users, meters, and assign officers to consumers

Key features:
- Real-time power consumption dashboard with live kW, V, A readings
- Interactive analytics charts (Day/Week/Month/Year views)
- Remote appliance control for Lights (1,2,3) and Fans (1,2)
- Smart alert notifications (anomaly, overage, bill milestones, idle auto-off)
- Automatic device scheduling for auto-OFF at set times
- Bill payment via Razorpay (UPI, cards, net banking)
- Firebase Authentication (Email/Password + Google Sign-In)
- Role-based routing and access control

Major objectives of WattX:
- Provide low-cost, real-time visibility of household energy usage for
  ordinary consumers using only a smartphone and a compact IoT meter.
- Help users actively reduce their electricity bill by giving actionable
  analytics, budget alerts, and energy-saving tips instead of just raw data.
- Offer a unified platform for monitoring, control, billing, and complaints
  so that users do not need separate apps or manual visits to KSEB offices.
- Demonstrate a complete IoT stack (hardware + MQTT + cloud backend + mobile
  app) that can be extended to solar and smart home scenarios.

Scope of the project (what WattX covers):
- Single-phase domestic consumers connected to KSEB, using one smart meter
  (METER001) per household in the prototype.
- Monitoring of instantaneous parameters (power, voltage, current, power
  factor, frequency) and aggregated kWh per day/week/month/year.
- Control of up to five relay-based loads (three lights and two fans).
- Basic billing simulation with configurable tariff slabs and integration
  to Razorpay in test mode for secure digital payments.
- Role-based dashboards for three roles (User, KSEB Officer, Admin) covering
  day-to-day operations like complaints, notices, and user management.

Items outside current scope (but possible in future):
- Three-phase industrial meters, detailed tariff integration with official
  KSEB APIs, and direct integration into KSEB production systems.
- Native iOS application and web portal (current prototype targets Android).
- Advanced AI-based non-intrusive load monitoring (NILM) for automatic
  appliance identification.

1.2 PROJECT SPECIFICATION

WattX is a mobile-based Smart Energy Monitoring System developed to connect
domestic consumers, smart meters, KSEB officers, and administrators on a
single integrated platform. The project focuses on real-time electricity
monitoring, remote appliance control, bill estimation and payment, and
digital management of complaints and notices, using a low-cost IoT meter
and a cloud-backed Flutter application.

Technology Stack Used
• Frontend: Flutter (Dart) Android application
• Backend: Firebase Realtime Database and Cloud Firestore
• Authentication: Firebase Authentication (Email/Password, Google Sign-In)
• IoT Bridge: Python 3 (mqtt_sync.py) with paho-mqtt
• Messaging: MQTT protocol via HiveMQ Cloud broker (TLS, port 8883)
• Payment Gateway: Razorpay Flutter SDK (test mode in prototype)
• Charts & UI: fl_chart, Google Fonts (Inter), Provider for state management
• Tools & IDE: Android Studio / VS Code, Git & GitHub for version control

User Roles
• Admin – Manages all users and smart meters, assigns KSEB officers to
  consumers, configures tariff slabs, monitors alerts and usage trends,
  and oversees overall system health from the admin dashboard.
• KSEB Officer – Views assigned consumers, tracks their complaints and
  service requests, updates complaint status, and publishes notices or
  announcements relevant to their service area.
• User (Consumer) – Registers and logs in to the app, views real-time
  energy usage, controls connected appliances, checks analytics charts,
  views and pays simulated electricity bills, manages schedules, and
  submits complaints or feedback.

Key Modules
• Authentication & Role-Based Access Control
• Real-Time Dashboard & IoT Device Monitoring
• Appliance Control & Scheduling (Lights and Fans)
• Energy Analytics & Bill Estimation
• Smart Notifications & Anomaly Alerts
• User Profile & Settings (including dark mode)
• Admin Dashboard (user and meter management, tariff configuration)
• KSEB Officer Dashboard (consumer list, complaints, notices)

Security & Performance
• Secure login and session handling using Firebase Authentication tokens.
• Role-based authorization enforced in the app and Firebase Security Rules
  so that users can only access their own data, while Admin and KSEB
  Officer roles have controlled elevated privileges.
• Encrypted communication via HTTPS for Firebase services and TLS-secured
  MQTT connections to HiveMQ Cloud and Razorpay payment gateway.
• Centralized cloud database with input validation and data sanitization
  for user forms, device commands, and billing data.
• Optimized real-time listeners and lightweight UI design to ensure smooth
  performance on low- and mid-range Android devices.

WattX thus provides a secure, scalable, and cost-effective smart energy
monitoring solution that improves household energy awareness, reduces
manual processes, and enhances coordination between consumers and KSEB
through a unified digital platform.

CHAPTER 2 - SYSTEM STUDY
==============================================================

2.1 INTRODUCTION

System study involves analyzing the present method of household electricity
monitoring and billing, identifying its limitations, and designing a better
digital solution. In WattX, the study focuses on understanding how domestic
consumers currently read their meters, receive bills, manage energy usage,
and register complaints with KSEB, and how these activities can be brought
together into a single smart energy monitoring platform.

2.2 EXISTING SYSTEM

In most households, electricity usage is tracked using traditional meters
installed by KSEB. Meter readings are taken manually once in a billing
cycle, and consumers receive only a final bill amount without real-time
feedback or detailed analytics. Complaints and service requests are usually
handled through physical visits or phone calls, and there is no integrated
system for monitoring usage, paying bills, and managing complaints from a
single interface.

2.2.1 NATURAL SYSTEM STUDIED

The natural system studied is the conventional electricity metering and
billing process where cumulative kWh readings, bill generation, complaint
registration, and basic consumer interactions with KSEB are managed using
paper-based records, manual meter reading, and separate portals or offices
without real-time visibility of consumption.

2.2.2 DESIGNED SYSTEM STUDIED

Some commercial smart home energy monitors and pilot smart meters by KSEB
were also studied. These systems provide partial digital features such as
online bill viewing or basic usage graphs, but they often lack full
integration with IoT smart meters, real-time appliance control, anomaly
alerts, or role-based dashboards for consumers, officers, and admins in the
local Kerala context.

2.3 DRAWBACKS OF EXISTING SYSTEM

- No real-time electricity consumption data available to consumers
- No appliance-level monitoring or remote control capability
- Monthly billing only with no mid-cycle alerts for overage usage
- No analytics charts or trend visualization for consumers
- Manual meter reading process is error-prone and delayed by weeks
- No integration with digital payment gateways for bill payment
- Consumers cannot schedule appliances to auto-turn-off
- No anomaly detection or smart energy-saving suggestions
- KSEB complaint and service request handling is not digitized

2.4 PROPOSED SYSTEM

The proposed system, WattX Smart Energy Monitor, is a fully integrated
IoT-based application that connects smart meters, domestic consumers, KSEB
officers, and admins on a single digital platform. It uses an ESP8266/ESP32
based smart meter with a PZEM-004T sensor to measure electrical parameters
in real time and publish them via MQTT to the cloud. A Flutter Android app
reads the live data from Firebase, displays real-time dashboards and
analytics charts, provides appliance control and scheduling, generates
estimated bills with Razorpay payment support, manages complaints and
notices, and offers role-based dashboards for users, officers, and admins.

2.5 ADVANTAGES OF PROPOSED SYSTEM

- Real-time electricity monitoring with sub-2-second update latency
- Mobile-first glassmorphism design with intuitive UX
- Remote appliance control from anywhere via mobile app
- Smart anomaly detection comparing consumption to 7-day rolling average
- Mid-cycle overage alerts to prevent bill shock at month-end
- Interactive line and bar charts for Day/Week/Month/Year energy trends
- Integrated Razorpay payment (UPI, cards, net banking, wallets)
- Auto scheduling to switch off devices at user-defined times
- Role-based access system for Users, KSEB Officers, and Admins
- Secure Firebase Authentication with Google Sign-In support

These advantages, combined with a cloud-native architecture using Firebase
and a managed MQTT broker, reduce manual effort, minimize errors, and
improve overall energy awareness and service quality for KSEB domestic
consumers.

3.1.4 FEASIBILITY STUDY QUESTIONNAIRE

1. How do you currently monitor household electricity usage?
Electricity usage is generally checked only through the monthly meter
reading or final electricity bill. This makes it difficult for users to
understand daily consumption patterns or identify sudden increases in usage.

2. What difficulties do you face in tracking electricity consumption?
Consumers often do not know how much power is being used at a specific time
or which periods lead to higher bills. This causes poor energy planning,
unexpected bill amounts, and limited awareness of wastage.

3. How are electricity bills currently checked and paid?
Bills are usually viewed through printed statements, utility portals, or
manual visits to payment centers. This process is less convenient and does
not provide continuous visibility into ongoing consumption before bill
generation.

4. What issues arise when trying to control household devices efficiently?
Lights and fans are normally controlled only through physical switches,
which means users cannot manage them remotely or schedule automatic switch
off. This often results in unnecessary energy consumption.

5. How are unusual consumption spikes identified at present?
Most users identify high consumption only after receiving the monthly bill.
There is no immediate alert mechanism to warn them about abnormal usage,
faulty devices, or over-limit consumption.

6. What information is needed in a smart energy monitoring system?
Users need live values such as power, voltage, current, frequency, power
factor, total usage trends, bill estimation, alert notifications, and basic
device control status in one place.

7. How are complaints or service issues currently reported to KSEB?
Complaints are usually reported through office visits, phone calls, or
separate utility channels. This can be slow, difficult to track, and not
directly linked to the user’s smart meter or app usage details.

8. What type of monitoring and reports would be useful for administrators?
Administrators require reports on user activity, smart meter status,
complaints, alerts, and overall usage trends so they can monitor the system
and improve service efficiency.

9. What challenges exist in managing energy usage manually?
Manual monitoring makes it difficult to compare daily or monthly patterns,
identify waste, or take preventive action before the bill increases.
Consumers are forced to react late instead of managing energy proactively.

10. How important is secure access control in a smart energy application?
It is highly important. Different roles such as consumer, KSEB officer, and
admin need different access levels to protect personal data, billing data,
and administrative controls from unauthorized access.

11. What improvements would a centralized digital system bring?
A centralized platform would provide continuous usage visibility, quicker
complaint handling, digital billing support, better communication with KSEB,
and more efficient energy management for households.

12. What technology features would help users most?
Real-time notifications, analytics dashboards, usage history, smart alerts,
remote appliance control, scheduling, and digital payment options would
significantly improve user convenience and energy awareness.

3.3 SOFTWARE DESCRIPTION

3.3.1 FLUTTER

Flutter is Google's open-source UI toolkit for building natively compiled

Flutter is used to build the Android mobile application.

Key advantages used in WattX:

Flutter version: 3.x | Dart SDK: >=3.0.0 <4.0.0

3.3.2 FIREBASE

Firebase is Google's cloud-based Backend-as-a-Service. WattX uses:

FIREBASE REALTIME DATABASE (firebase_database ^10.4.0):
- NoSQL JSON-tree database for real-time IoT data
- Stores live energy readings from the smart meter
- Stores device states (lights/fans ON/OFF) for control sync
- Stores analytics summaries (day/week/month/year)
- Stores device schedules and smart alerts
- Real-time onValue stream listeners update Flutter UI instantly

Database structure:
  /readings/METER001/latest       <- live IoT reading snapshot
  /devices/METER001/controls      <- device toggle states (light1..fan2)
  /analytics/METER001/day         <- daily energy data
  /analytics/METER001/week        <- weekly energy data
  /analytics/METER001/month       <- monthly energy data
  /analytics/METER001/year        <- yearly energy data
  /schedules/{id}                 <- auto-OFF device schedules
  /alerts/{userId}/{alertId}      <- smart notifications

CLOUD FIRESTORE (cloud_firestore ^4.14.0):
- Stores user profiles, complaints, bills, payments, notices,
  service requests in structured document collections
- Supports complex queries for admin and KSEB officer dashboards

FIREBASE AUTHENTICATION (firebase_auth ^4.17.0):
- Email/Password sign-in with form validation
- Google Sign-In with forceAccountSelection: true
- Password reset via email link
- Re-authentication before sensitive profile changes

SECURITY AND PRIVACY CONSIDERATIONS:
- All communication between the app and Firebase uses HTTPS/TLS, and the
  MQTT connection to HiveMQ Cloud is protected with TLS on port 8883.
- Firebase Security Rules ensure that users can access only their own data
  (profiles, alerts, bills); Admin and KSEB Officer roles have additional
  permissions but are still restricted from arbitrary writes.
- Authentication tokens and password handling are delegated to Firebase
  Auth, reducing the risk of implementing insecure custom auth logic.
- No card or UPI details are stored in WattX. Razorpay handles sensitive
  payment information and returns only secure transaction identifiers that
  are stored in Firestore for billing history.

4.4 DATABASE DESIGN

4.4.1 RDBMS (Relational Conceptual Model)

WattX follows a structured cloud database design to store and manage smart
energy monitoring data in an organized manner. Firebase Realtime Database
and Cloud Firestore are used as the backend databases for maintaining user
profiles, smart meter details, live energy readings, device control states,
alerts, bills, complaints, schedules, notices, and administrative data.

The database follows a logical relational structure where the major modules
of the system are connected through unique identifiers so that data remains
consistent and duplication is minimized. Although Firebase is a NoSQL
platform, the system is modeled conceptually like an RDBMS for design and
documentation purposes.

Each major module of WattX such as User Management, Meter Management,
Reading Collection, Alerts, Billing, Complaint Handling, Scheduling, and
Administration has separate collections or nodes that are linked together.

For example, the users collection stores the login and profile details of
all users such as Admin, KSEB Officer, and Consumer. The meters collection
is linked to users through owner identifiers. The readings node is linked
to a specific smart meter. The alerts, bills, complaints, and schedules are
associated with users and meters through IDs. This structure helps WattX
maintain data integrity, efficient access, and proper coordination between
consumer, officer, and admin modules.

4.4.2 NORMALIZATION

Normalization is the process of organizing data to reduce redundancy and
maintain consistency. In WattX, normalization principles are applied in the
database design to ensure efficient storage and proper relationships between
different modules.

First Normal Form (1NF)
In 1NF, all records contain atomic values and no repeating groups.
Example: In the readings data, each record stores a single set of values
such as power, voltage, current, frequency, power factor, and timestamp for
one meter reading. Multiple readings are stored as separate records.

Second Normal Form (2NF)
In 2NF, the structure must already satisfy 1NF and all non-key attributes
must depend fully on the primary key.
Example: In the bills collection, fields such as month, year, totalKWh,
amount, and dueDate depend entirely on the bill ID and are not partially
dependent on other fields.

Third Normal Form (3NF)
In 3NF, the structure must already satisfy 2NF and there should be no
transitive dependency, meaning non-key attributes should not depend on
other non-key attributes.
Example: User role and profile information are stored directly in the users
collection, while meter-specific details such as firmwareVersion and status
are stored separately in the meters collection to avoid redundancy.

By applying normalization, WattX reduces unnecessary duplication of user,
meter, billing, and alert data and improves maintainability of the overall
system.

4.4.3 SANITIZATION

In WattX, sanitization is implemented to ensure secure handling of user
inputs and to protect personal, billing, and device-related data from
security risks. Since the system allows users, KSEB officers, and admins to
enter data such as login credentials, profile details, complaint messages,
device schedules, and billing-related information, all inputs are validated
and sanitized before being stored in Firebase.

The following sanitization measures are applied in WattX:
- Client-side input validation using Flutter form validators
- Email validation and trimming before authentication and storage
- Name, address, and complaint fields checked for empty or invalid input
- Numeric validation for bill amounts, usage values, and threshold limits
- Boolean-only validation for device control states written to RTDB
- Firebase Authentication for secure password handling and session control
- Firebase Security Rules to prevent unauthorized reads and writes

By implementing sanitization, WattX ensures secure cloud data storage,
protects user information, prevents invalid or malicious input, and helps
maintain the reliability and integrity of the smart energy monitoring
system.

4.4.4 INDEXING

Indexing in WattX is used to improve database performance and speed up data
retrieval. Firebase Realtime Database and Cloud Firestore support indexed
lookups that help the application quickly access frequently used records
such as user profiles, meter details, bills, complaints, alerts, and
analytics data.

Primary identifiers such as uid, meterId, billId, and alertId are used for
efficient access to important records. In Firestore, composite indexes are
applied on commonly queried fields such as role, userId, month, year,
timestamp, and isRead to improve search and filtering operations.

Examples of indexing used in WattX:
- users: role (for admin filtering of user types)
- complaints: userId + timestamp (for complaint history)
- bills: userId + month + year (for monthly bill lookup)
- alerts: userId + isRead + timestamp (for notification display)

Thus, indexing helps in faster queries, smoother dashboard loading,
efficient filtering, and better overall performance of the WattX system.

    that relays on the ESP8266 board switch accordingly.
  - SchedulesScreen allows users to create, list, and delete auto-OFF rules
    stored under /schedules; a periodic server-side job checks due times and
    updates device states to enforce schedules.

Step 7: Analytics and Billing Logic

  - Implemented a simple tariff engine that multiplies kWh by slab-wise
    rates to compute an estimated bill, stored as BillModel objects.

    bill document is updated with isPaid=true and a transaction reference.

-- Firebase RTDB live listener (dashboard_screen.dart excerpt) --
void _listenToLiveReadings() {
  FirebaseDatabase.instance
    .ref('readings/METER001/latest')
    .onValue
    .listen((event) {
      if (event.snapshot.exists) {
        final data = Map<dynamic,dynamic>.from(event.snapshot.value as Map);
        setState(() {
          _currentPower   = (data['power']   ?? 0.0).toDouble();
          _currentVoltage = (data['voltage'] ?? 0.0).toDouble();
          _currentCurrent = (data['current'] ?? 0.0).toDouble();
        });
      }
    });
}

-- Device toggle (database_service.dart excerpt) --
Future<void> toggleDevice(String deviceKey, bool value) async {
  await FirebaseDatabase.instance
    .ref('devices/METER001/controls/$deviceKey')
    .set(value);
}

-- Anomaly detection (anomaly_service.dart excerpt) --
String detectAnomaly(double currentPower, double avg7day) {
  if (avg7day == 0) return 'normal';
  double pct = ((currentPower - avg7day) / avg7day) * 100;
  if (pct > 100) return 'critical';
  if (pct > 50)  return 'high';
  if (pct < -20) return 'efficient';
  return 'normal';
}

-- mqtt_sync.py (IoT-Firebase bridge excerpt) --
import paho.mqtt.client as mqtt
import firebase_admin
from firebase_admin import credentials, db
import json

cred = credentials.Certificate("serviceAccountKey.json")
firebase_admin.initialize_app(cred, {
    'databaseURL': 'https://your-project-default-rtdb.firebaseio.com'
})

def on_message(client, userdata, msg):
    payload = json.loads(msg.payload.decode())
    db.reference('readings/METER001/latest').set({
        'power'   : payload.get('power', 0),
        'voltage' : payload.get('voltage', 0),
        'current' : payload.get('current', 0),
        'timestamp': payload.get('timestamp', 0)
    })

client = mqtt.Client()
client.username_pw_set("mqttuser", "mqttpassword")
client.tls_set()
client.connect("your-cluster.hivemq.cloud", 8883)
client.subscribe("METER001/readings")
client.on_message = on_message
client.loop_forever()
