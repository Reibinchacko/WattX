ABSTRACT
------------------------------------------------------------

WattX is an IoT-based smart energy monitoring mobile application developed
using Flutter and Firebase. It enables real-time monitoring of household
electricity consumption by integrating IoT smart meters via MQTT protocol.
The application provides live metrics including power (kW), voltage (V),
current (A), power factor, and frequency. Users can remotely control home
appliances (lights and fans), view interactive analytics charts, receive
smart alerts for anomalies and overage usage, schedule automatic device
shutdowns, and pay electricity bills via Razorpay. The system supports
role-based access for Users, KSEB Officers, and Admins. Firebase Realtime
Database enables real-time synchronization between the IoT device and app.

CHAPTER 1 - INTRODUCTION
==============================================================

1.1 PROJECT OVERVIEW

WattX is a smart energy monitoring system that leverages IoT and mobile
technology to give homeowners real-time visibility and control over their
electricity consumption. The application consists of two key components:

(1) An IoT smart energy meter (hardware) that measures electrical parameters
    such as power (kW), voltage (V), current (A), power factor, and frequency
    in real time using an ESP8266 microcontroller and PZEM-004T sensor.

(2) A Flutter-based Android mobile application that connects to the IoT
    device via Firebase Realtime Database and MQTT protocol to display live
    data, provide analytics, enable device control, and send smart alerts.

The system supports three types of users:
- Regular Users: Monitor energy, control appliances, view bills, receive alerts
- KSEB Officers: Manage consumer complaints, service requests, post notices
- Admins: Manage all users, meters, and assign officers to consumers

Key features:
- Real-time power consumption dashboard with live kW, V, A readings
- Interactive analytics charts (Day/Week/Month/Year views)
- Remote appliance control for Lights (1,2,3) and Fans (1,2)
- Smart alert notifications (anomaly, overage, bill milestones, idle auto-off)
- Automatic device scheduling for auto-OFF at set times
- Bill payment via Razorpay (UPI, cards, net banking)
- Firebase Authentication (Email/Password + Google Sign-In)
- Role-based routing and access control

CHAPTER 2 - SYSTEM STUDY
==============================================================

2.1 INTRODUCTION

A system study analyzes the needs and problems of existing energy monitoring
systems, identifies their drawbacks, and designs an improved solution. Energy
consumption management is critical today due to rising electricity tariffs
and increased environmental awareness. Traditional electricity meters only
show cumulative kWh consumption with no real-time feedback, analytics, or
remote control. This gap is exactly what WattX aims to fill.

2.2 EXISTING SYSTEM

2.2.1 NATURAL SYSTEM STUDIED

The conventional household electricity metering system consists of an
electromechanical or digital energy meter installed by the utility board
(KSEB). The meter records cumulative energy in kWh. A meter reader visits
the household monthly to record readings and bills are generated manually.
There is no real-time data, no remote monitoring, and no mobile interface.
Consumers receive bill statements via post or utility portals showing only
monthly totals, with no way to know per-appliance consumption or get alerts
when usage crosses a threshold.

2.2.2 DESIGNED SYSTEM STUDIED

Several commercial smart home energy monitors exist:
- Sense Home Energy Monitor: Uses ML to identify appliances but is expensive
  (USD 300+) and not available in India with KSEB integration.
- Shelly EM: European IoT energy monitor with cloud integration but lacks
  a consumer-facing mobile app in Malayalam/local context.
- KSEB Smart Meters (pilot): Rolling out in select areas of Kerala but
  without a consumer-facing mobile application for real-time monitoring.

These systems lack affordability, local availability, KSEB integration,
or mobile-first design for Indian consumers.

2.3 DRAWBACKS OF EXISTING SYSTEM

- No real-time electricity consumption data available to consumers
- No appliance-level monitoring or remote control capability
- Monthly billing only with no mid-cycle alerts for overage usage
- No analytics charts or trend visualization for consumers
- Manual meter reading process is error-prone and delayed by weeks
- No integration with digital payment gateways for bill payment
- Consumers cannot schedule appliances to auto-turn-off
- No anomaly detection or smart energy-saving suggestions
- KSEB complaint and service request handling is not digitized

2.4 PROPOSED SYSTEM

WattX is a comprehensive IoT-based smart energy monitoring solution tailored
for Indian households connected to KSEB.

HARDWARE COMPONENT:
A smart IoT energy meter based on an ESP8266/ESP32 microcontroller with a
PZEM-004T energy sensor that measures voltage, current, power, and power
factor in real time. The device publishes readings to an MQTT broker every
2 seconds with meter ID METER001.

SOFTWARE COMPONENT:
A Flutter Android application connected to Firebase Realtime Database.
A Python MQTT bridge script (mqtt_sync.py) subscribes to the MQTT broker
and writes incoming IoT data to Firebase. The mobile app reads from Firebase
in real-time and displays live metrics.

KEY MODULES:
1. Dashboard        : Real-time kW, voltage, current, frequency, power factor
2. Analytics        : Charts with Day/Week/Month/Year filter tabs
3. Control Center   : Toggle lights and fans with live IoT sync
4. Schedules        : Set auto-OFF device schedules
5. Bill & Savings   : View monthly bill, pay via Razorpay, savings tips
6. Notifications    : Smart alerts for anomalies, overage, idle auto-off
7. Profile          : Edit name, email, password, dark mode toggle
8. Admin Dashboard  : Manage users, meters, assign officers
9. KSEB Officer     : Handle complaints, service requests, notices

2.5 ADVANTAGES OF PROPOSED SYSTEM

- Real-time electricity monitoring with sub-2-second update latency
- Mobile-first glassmorphism design with intuitive UX
- Remote appliance control from anywhere via mobile app
- Smart anomaly detection comparing consumption to 7-day rolling average
- Mid-cycle overage alerts to prevent bill shock at month-end
- Interactive line and bar charts for Day/Week/Month/Year energy trends
- Integrated Razorpay payment (UPI, cards, net banking, wallets)
- Auto scheduling to switch off devices at user-defined times
- Role-based access system for Users, KSEB Officers, and Admins
- Secure Firebase Authentication with Google Sign-In support

4.2.2 SEQUENCE DIAGRAM

LOGIN FLOW:
1. User enters email + password on LoginScreen
2. LoginScreen calls AuthService.signIn(email, password)
3. AuthService calls FirebaseAuth.signInWithEmailAndPassword()
4. Firebase Auth returns UserCredential on success
5. AuthService reads user role from Cloud Firestore /users/{uid}
6. Role-based navigation:
   user -> HomeScreen (Dashboard)
   admin -> AdminDashboardScreen
   kseb_officer -> KSEBOfficerDashboardScreen

REAL-TIME DASHBOARD UPDATE:
1. ESP8266 measures power=1.42kW, voltage=231V, current=6.14A
2. ESP8266 publishes to MQTT topic: "METER001/readings"
3. HiveMQ Cloud broker receives and forwards message
4. mqtt_sync.py (Python bridge) receives message via paho-mqtt
5. mqtt_sync.py writes to Firebase /readings/METER001/latest
6. Flutter DashboardScreen onValue listener fires
7. Widget rebuilds with new values: Power=1.42kW, V=231, A=6.14

7. ESP8266 publishes confirmation on METER001/status
8. mqtt_sync.py writes status back to Firebase RTDB
9. ControlScreen active count badge updates: "1/5 ON"

4.4 DATABASE DESIGN

4.4.1 RDBMS (Relational Conceptual Model)

Logical relationships (Firebase NoSQL but modeled relationally):
  Users      (1) -----> (1)   Meters
  Meters     (1) -----> (N)   Readings
  Users      (1) -----> (N)   Alerts
  Users      (1) -----> (N)   Bills
  Users      (1) -----> (N)   Complaints
  Users      (1) -----> (N)   Schedules
  KSEB Offcr (1) -----> (N)   Consumers (Users)
  Admins     (1) -----> (N)   All Users

4.4.2 NORMALIZATION

1NF: All fields contain atomic (single) values. No repeating groups.
     Each reading has exactly one value per field (power, voltage, etc.)

2NF: All non-key attributes depend fully on the primary key (document ID).
     In the users collection, all fields (name, email, role) depend on uid.

3NF: No transitive dependencies. User role is stored directly in the
     users document without indirect dependencies on other tables.

4.4.3 SANITIZATION

All user inputs are sanitized before storage:
- Email: trimmed + validated with RegExp pattern
- Password: minimum 6 characters enforced client-side
- Name: trimmed, empty check before saving to Firestore
- Payment amounts: validated as positive double values
- Device toggle values: boolean-only written to Firebase RTDB
- Firebase Security Rules enforce server-side access control

4.4.4 INDEXING

Firebase RTDB indexes on JSON key paths automatically.
Cloud Firestore composite indexes used:
  - users: role (for admin user-type queries)
  - complaints: userId + timestamp (for user complaint history)
  - bills: userId + month + year (for monthly bill lookup)
  - alerts: userId + isRead + timestamp (for notification list)

void _listenToLiveReadings() {
  FirebaseDatabase.instance
    .ref('readings/METER001/latest')
    .onValue
    .listen((event) {
      if (event.snapshot.exists) {
        final data = Map<dynamic,dynamic>.from(event.snapshot.value as Map);
        setState(() {
          _currentPower   = (data['power']   ?? 0.0).toDouble();
          _currentVoltage = (data['voltage'] ?? 0.0).toDouble();
          _currentCurrent = (data['current'] ?? 0.0).toDouble();
        });
      }
    });
}

-- Device toggle (database_service.dart excerpt) --
Future<void> toggleDevice(String deviceKey, bool value) async {
  await FirebaseDatabase.instance
    .ref('devices/METER001/controls/$deviceKey')
    .set(value);
}

-- Anomaly detection (anomaly_service.dart excerpt) --
String detectAnomaly(double currentPower, double avg7day) {
  if (avg7day == 0) return 'normal';
  double pct = ((currentPower - avg7day) / avg7day) * 100;
  if (pct > 100) return 'critical';
  if (pct > 50)  return 'high';
  if (pct < -20) return 'efficient';
  return 'normal';
}

-- mqtt_sync.py (IoT-Firebase bridge excerpt) --
import paho.mqtt.client as mqtt
import firebase_admin
from firebase_admin import credentials, db
import json

cred = credentials.Certificate("serviceAccountKey.json")
firebase_admin.initialize_app(cred, {
    'databaseURL': 'https://your-project-default-rtdb.firebaseio.com'
})

def on_message(client, userdata, msg):
    payload = json.loads(msg.payload.decode())
    db.reference('readings/METER001/latest').set({
        'power'   : payload.get('power', 0),
        'voltage' : payload.get('voltage', 0),
        'current' : payload.get('current', 0),
        'timestamp': payload.get('timestamp', 0)
    })

client = mqtt.Client()
client.username_pw_set("mqttuser", "mqttpassword")
client.tls_set()
client.connect("your-cluster.hivemq.cloud", 8883)
client.subscribe("METER001/readings")
client.on_message = on_message
client.loop_forever()
