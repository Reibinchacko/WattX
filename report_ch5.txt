CHAPTER 5 - SYSTEM TESTING

5.1 INTRODUCTION

System testing is a critical and structured phase of the Software Development
Life Cycle (SDLC) that evaluates the complete, integrated system to verify
that it meets all specified functional and non-functional requirements. Unlike
unit testing (which isolates individual functions) or integration testing
(which tests component interactions), system testing treats the entire WattX
platform - Flutter app, Firebase backend, MQTT bridge, and IoT hardware - as
a single black-box system and validates it against the requirements defined
in Chapter 3.

For WattX, system testing is particularly important because the application
involves multiple interconnected components operating in real time:
  - IoT hardware publishing sensor data every 2 seconds
  - MQTT broker relaying data between IoT and cloud
  - Python bridge writing data to Firebase Realtime Database
  - Flutter app reading Firebase streams and updating UI
  - Firebase Realtime Database storing user, device, bill, and complaint data
  - Razorpay processing real payment transactions (in test mode)
  - Firebase Authentication managing secure user sessions

A failure in any one of these components can cascade into system-wide
failures. Comprehensive testing across all layers is therefore essential
to ensure WattX operates reliably in production.

TESTING OBJECTIVES:
  1. Verify that live IoT data from the smart meter appears correctly in
     the Flutter dashboard within a 2-3 second latency window
  2. Confirm that device toggle commands from the app reach the physical
     hardware relay within 2 seconds via Firebase -> MQTT -> ESP8266
  3. Validate that Firebase Authentication correctly handles login, signup,
     Google Sign-In, role assignment, and password reset flows
  4. Verify role-based routing: users go to Dashboard, admins to Admin
     Dashboard, KSEB Officers to Officer Dashboard
  5. Confirm that analytics charts correctly aggregate and display energy
     data for Day, Week, Month, and Year views
  6. Verify that smart alerts (anomaly, overage, idle auto-off, bill
     milestone) are generated under the defined threshold conditions
  7. Validate that device schedules execute auto-OFF at the correct time
  8. Confirm that Razorpay payment flow works end-to-end in test mode
  9. Verify that complaint submission, KSEB Officer management, and admin
     user management functions work correctly
  10. Test UI responsiveness on different screen sizes (5", 6", 6.5" phones)
  11. Validate Python automation and Firebase Console verification steps

TESTING ENVIRONMENT:
  Development OS       : Windows 11 Home (64-bit)
  Flutter Version      : 3.x (stable channel)
  Dart SDK Version     : 3.x
  Android Emulator     : Pixel 6 API 34 (AVD Manager)

  Firebase Project     : wattx-16024 (Firebase Realtime Database)

5.2 TEST PLAN

The WattX test plan defines the scope, strategy, resources, and schedule for
all testing activities. Testing follows a bottom-up approach: individual
components are tested first (unit testing), then their interactions
(integration testing), then the full system (validation testing), then user
acceptance (UAT), and finally automated regression testing.

TEST PLAN OVERVIEW:

| Test Level          | Focus Area                      | Tools Used        |
|---------------------|---------------------------------|-------------------|
| Unit Testing        | Core Dart models                | flutter_test      |
| Integration Testing | Multi-component workflows       | Manual + print log|
| Validation Testing  | Full system against requirements| Manual scenario   |
| UAT                 | Real user acceptance            | 10 participants   |
| Automation Testing  | Android app smoke regression    | Appium + UiAutomator2 |
| Backend Verification| Firebase and bridge validation  | Python scripts + Firebase Console |

ENTRY CRITERIA (before testing begins):
  - All Flutter screens are coded and compile without errors
  - Firebase project is configured (RTDB, Firestore, Auth enabled)
  - mqtt_sync.py connects to HiveMQ and Firebase without errors
  - IoT hardware (ESP8266 + PZEM-004T) is powered and connected to WiFi
  - At least one test user account is registered in Firebase Auth

EXIT CRITERIA (testing is complete when):
  - All automation and backend verification scripts complete without failure
  - End-to-end payment flow completes in Razorpay test mode
  - No CRITICAL or HIGH severity bugs remain open
  - UAT satisfaction rating >= 4.0/5.0

5.3 DEFECT SUMMARY

The testing activities in this chapter identified mostly minor usability and
stability issues rather than critical functional failures. The main findings
are summarised below:

  - Small-screen layout clipping was observed in analytics labels and was
    corrected by adjusting padding and text size.
  - Google Sign-In account selection behaved inconsistently on an older
    Android 9 device and was improved through explicit account selection.
  - Scheduling is implemented using in-memory timers; therefore schedules are
    reliable only while the app remains active. This is acceptable for the
    current prototype but should be upgraded in future work using persistent
    background scheduling.
  - Automation coverage currently focuses on smoke testing. Dedicated Dart
    unit and widget tests should be added in future iterations for stronger
    regression protection.

5.4 CHAPTER SUMMARY

This chapter presented the complete testing strategy for WattX, covering
code-level verification, integration testing, full-system validation, user
acceptance testing, Python/Appium automation, and Firebase backend checks.
Across the documented tests, the system satisfied the major project
requirements: live data reached the dashboard correctly, device controls were
propagated through Firebase and MQTT, authentication and role-based access
worked correctly, alerts and schedules operated as expected, and users rated
the application positively during UAT.

Overall, the Chapter 5 results demonstrate that WattX is functionally viable
as a smart energy monitoring prototype. The remaining work is not core defect
repair but hardening: expanding automated Dart tests, improving persistent
scheduling, and broadening device coverage for production readiness.
