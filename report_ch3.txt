system is technically achievable, economically justifiable, and behaviorally
acceptable by its intended users. For WattX, a three-dimensional feasibility
analysis was conducted before committing to full development.

3.1.2 TECHNICAL FEASIBILITY

Technical feasibility assesses whether the technology required for WattX
exists, is mature, and is within the development team's capability to use.

TECHNOLOGY MATURITY ASSESSMENT:

Flutter & Dart:
  - Released by Google in 2018 (stable), actively maintained
  - Version 3.x (2022 onwards) with full null safety
  - Used by 500,000+ developers globally
  - Powers apps like Google Pay India, Alibaba Xianyu, BMW app
  - 150,000+ GitHub stars (one of the most starred frameworks)
  - Dart language has strong typing, async/await, and compiles to native ARM
  - Verdict: MATURE and widely proven in production

Firebase Realtime Database:
  - Google service since 2012 (acquired from Firebase Inc.)
  - 99.95% uptime SLA with global CDN distribution
  - Sub-100ms read latency with onValue stream listeners
  - Used by millions of apps globally (Shazam, Lyft early versions, etc.)
  - Free Spark plan supports up to 100 simultaneous connections
  - Verdict: MATURE, enterprise-grade, and proven for IoT applications

Cloud Firestore:
  - GA since 2019, built by Google as RTDB's successor
  - Multi-region replication (Nam5 region for WattX)
  - Supports complex queries, compound indexes, real-time listeners
  - Verdict: MATURE and ideal for structured user/admin data

MQTT Protocol:
  - Developed in 1999 by IBM for satellite telemetry, now ISO standard
  - ISO/IEC 20922:2016 standard
  - Used by Facebook Messenger (scaling IoT), Amazon IoT, Microsoft Azure IoT
  - PAHO library (Eclipse Foundation) is the gold standard Python client
  - HiveMQ is one of the world's most reliable MQTT brokers (enterprise-grade)
  - Verdict: MATURE, proven IoT messaging standard

ESP8266 Microcontroller:
  - Espressif Systems ESP8266EX chipset, released 2014
  - WiFi SoC with TCP/IP stack, 80 MHz CPU, 4 MB flash
  - Arduino/MicroPython ecosystem with millions of projects
  - PZEM-004T integration well-documented in community
  - Verdict: MATURE, combat-tested in IoT projects worldwide

Razorpay:
  - India's leading payment gateway (founded 2014)
  - Processes millions of transactions daily
  - Supports UPI, RuPay, VISA, Mastercard, all major Indian banks
  - Flutter SDK v1.4.1 with well-documented integration
  - Used by Zomato, Swiggy, BookMyShow, OYO, Dunzo
  - Verdict: MATURE, trusted by major Indian companies

DEVELOPMENT TEAM CAPABILITY:
  - Prior experience with Flutter (completed working screens in Dart)
  - Firebase setup completed successfully (auth, RTDB, Firestore configured)

use, and benefit from the proposed system. This was assessed through a
structured survey conducted among 30 participants in Kanjirappally and
surrounding areas (Kottayam district, Kerala).

SURVEY METHODOLOGY:
  - Sample size: 30 participants
  - Demographics: College students (20), Household members aged 35-55 (10)

3.3 SOFTWARE DESCRIPTION

3.3.1 FLUTTER

Flutter is Google's open-source UI software development kit for building
natively compiled, multi-platform applications from a single codebase.
Created by Google and first released as beta in 2017, Flutter reached its
first stable version (1.0) in December 2018. As of 2025, Flutter 3.x
supports six platforms: Android, iOS, Web, Windows, Linux, and macOS.

CORE FLUTTER ARCHITECTURE:
Flutter applications are built using a reactive widget tree. Every element
visible on screen is a "widget" - an immutable description of part of the
user interface. Widgets are composed into trees, and when state changes,
Flutter efficiently rebuilds only the parts of the widget tree that changed
(using the "element tree" and "render tree" for diffing).

a) StatelessWidget: A widget that describes part of the UI that does not
   depend on any mutable state. Used in WattX for:
   - Static display components (labels, icons, metric cards)
   - Navigation routing destinations that receive all data via constructor

b) StatefulWidget: A widget that can change over time. Has an associated
   State object that persists across rebuilds. Used in WattX for:
   - DashboardScreen (updates when Firebase streams new readings)
   - ControlScreen (updates toggle states when Firebase changes)
   - AnalyticsScreen (updates when user taps a different time filter)
   - LoginScreen (manages form validation state)

PROVIDER STATE MANAGEMENT IN WATTX:
The Provider package (by Remi Rousselet) is an InheritedWidget wrapper that
makes state available throughout the widget tree without explicitly passing
it down via constructors.

WattX uses one main Provider:
  ThemeProvider extends ChangeNotifier:
    - Stores current ThemeMode (dark/light)
    - Loads persisted preference on startup via shared_preferences
    - Calls notifyListeners() when mode changes
    - All screens rebuild automatically when theme changes

Usage in main.dart:
  ChangeNotifierProvider(
    create: (_) => ThemeProvider(),
    child: WattXApp(),
  )

Access in any widget:
  final theme = Provider.of<ThemeProvider>(context);
  theme.toggleDarkMode(); // triggers app-wide theme change

FIREBASE AUTHENTICATION:
  Version: firebase_auth ^4.17.0
  
  Firebase Authentication provides backend services for secure user
  authentication. WattX uses two sign-in providers:
  
  a) Email/Password Authentication:
     - User creates account with name, email, and password (min 6 chars)
     - Password is hashed by Firebase (bcrypt/scrypt), never stored in plain
     - On successful signup: auth.createUserWithEmailAndPassword() creates
       an auth account AND a corresponding Firestore /users/{uid} document
       with the user's name, email, role='user', budgetLimit=100.0
     - On login: auth.signInWithEmailAndPassword() returns UserCredential
     - On forgot password: auth.sendPasswordResetEmail(email) sends a link
     - Re-authentication: auth.currentUser.reauthenticateWithCredential()
       is called before sensitive changes (email/password update)
  
  b) Google OAuth 2.0 Sign-In:
     - GoogleSignIn(scopes:['email']).signIn() opens native account selector
     - forceAccountSelection: true ensures account picker always shows
       (prevents silent sign-in with cached account)
     - Returns idToken and accessToken from Google
     - Combined into GoogleAuthProvider.credential and passed to Firebase:
       auth.signInWithCredential(googleCredential)
     - If new user: auto-creates Firestore /users/{uid} document with
       Google display name, photo URL, role='user'
  
  c) Role-Based Routing:
     After login, AuthService reads the user's role field from Firestore:
       final doc = await FirebaseFirestore.instance
           .collection('users')
           .doc(auth.currentUser!.uid)
           .get();
       final role = doc.data()!['role'];
     Navigation:
       if (role == 'admin') -> push AdminDashboardScreen
       if (role == 'kseb_officer') -> push KSEBOfficerDashboardScreen
       else -> push DashboardScreen (regular user home)

  d) Session Persistence:
     firebase_auth persists the auth session locally. On app restart,
     authStateChanges() stream immediately emits the last signed-in user.
     SplashScreen listens to this stream and routes accordingly, eliminating
     the need to sign in again on every app launch.
